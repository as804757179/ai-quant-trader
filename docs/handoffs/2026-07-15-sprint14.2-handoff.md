## 2. 项目目标与长期基线

项目最终目标是：在**不接入真实资金**的前提下，使用真实、可追踪、Point-in-Time 的 A 股数据，持续监控市场，完成可解释的自动模拟交易闭环：

```text
数据/事件 → 研究候选与持仓复评 → 组合决策 → 风控/授权 → 模拟订单与成交
→ 持仓/资金/费用/盈亏 → 收盘对账 → 盘后复盘 → 待人工批准的策略优化候选
```

长期设计基线是：

- [2026-07-13-continuous-full-market-paper-trading-design.md](../superpowers/specs/2026-07-13-continuous-full-market-paper-trading-design.md)

它不是已完成能力清单，也不证明盈利。核心原则：先验证数据和策略，再开放任何交易能力；AI 不能直接或间接创建订单；小样本或正确回测不代表持续盈利。

## 2.1 后续开发必须遵守的工作规则

### 复用优先，避免重复造轮子

1. **先调查，再实现。** 当前任务若已有成熟的开源实现、项目已有组件或标准库能力，先评估复用；不要因“实现简单”就重复造轮子。
2. **现成能力优先。** Provider 客户端、交易日历、解析器、重试、限流、序列化、表格、图表、通知、任务调度、审计等，优先使用项目已安装依赖或成熟、维护活跃的开源组件。
3. **新增依赖必须最小化。** 先确认许可证、维护活跃度、Windows 本地兼容性、安全更新、离线/降级行为、与现有依赖冲突及是否满足数据血缘要求；没有用户同意，不新增第三方包。
4. **只在无法安全复用时自建。** 自建前明确缺口、替代方案、维护成本、测试和回滚；自建实现必须小、可测试、可替换，不能变成隐藏的平台重写。
5. **不得把 Provider fallback 误当容灾。** 市场数据、公告、新闻、财务和交易路径中，来源变更必须显式记录并通过相应认证；禁止静默换源、Mock、Synthetic 或上一值补齐。

### 先设计确认，再做跨模块改造

1. 简单、局部、低风险任务：定向读取 → 最小修改 → 相关测试 → 简短结果。
2. 涉及数据库迁移、数据语义、回测、策略、风控、组合、订单、AI 到订单边界、发布权限或正式 Sprint：先说明目标、成功标准、最小方案和取舍；用户确认设计后再实施。
3. 复杂任务应提供可选方案及影响，不猜测会改变业务结果的需求。
4. 只修改当前范围所需文件；不顺便重构、不堆叠未确认模块、不删除“看似无用”的文件。
5. 发现无关问题只记录到 Backlog/报告，除非它直接阻塞当前验收。

### 验证与报告纪律

1. 结论必须来自实际命令、测试、接口或浏览器证据；无法验证即标记 `BLOCKED`，不得猜测 PASS。
2. 不得通过删除测试、skip、xfail、xpass、扩大容差、关闭门禁或伪造数据来获得 PASS。
3. 正式 Sprint 必须产出 Tracking Report，写清：实际完成/未完成、测试与验收结果、P0/P1/P2、安全开关和下一阶段准入结论。普通局部任务按用户要求提供简短结果或报告。
4. 修改影响长期数据、回测、交易或安全边界时，必须同步评估 ADR、测试、验收脚本和回滚方案；当前用户明确需求高于旧规则，但不能静默绕过关键安全能力。
5. 所有面向用户的时间统一使用 `Asia/Shanghai`，格式 `yyyy-MM-dd HH:mm:ss`；源码文本为 UTF-8。

### 数据、研究、交易与 AI 不可越过的边界

1. unknown、synthetic、uncertified、无来源或无时点证据的数据，不能进入可信研究、回测、选股、模拟成交或交易依据。
2. 不得使用未来可见的行情、公告、新闻、财报、企业行动或数据库记录；信号必须只使用当时公开且系统可得的信息。
3. AI 可以分析、解释、评分与生成 recommendation，但不能直接或间接创建订单、改变风控或自动修改生产策略。
4. 无数据、来源冲突、日历/证券状态不明、时效不达标或执行引用未授权时 fail closed；不以“为了模拟完整”产生假 K 线、假价格、假成交或假盈亏。
5. 回测、研究候选、模拟成交、Paper 和 Live 是不同权限层；上一层通过不自动授予下一层权限。

### 前端、运行与 Git 操作原则

1. 前端以当前确认的 Web 原型为准；无后端接口时显示中文“待接入”、真实空状态、失败状态或无权限状态，不伪造业务结果。当前优先 Web，不额外承诺移动端适配。
2. 页面必须显示数据来源、数据截止时间、版本和关联 ID；绿色仅表示通过，灰色表示不操作，橙色表示待审核，红色表示拒绝。
3. 不启动非标准常驻进程；使用项目 `start-local/start-dev/stop-local` 脚本，所有长命令设置超时并主动报告进度。
4. 保留用户和既有 Sprint 的未提交变更。未获明确授权不得 reset、clean、批量删除、强制覆盖、推送或创建 PR。
5. 搜索优先 `rg` 和精确调用链；不要每次全仓扫描、重读所有 ADR/Tracking Report 或重复生成项目总览。

## 10. 推荐的下一阶段：Sprint14.3 设计与实现顺序

建议先执行“**多维研究数据时点语义与认证交易日历覆盖**”的设计确认，再进入实现。不要跳到自动订单。

### 建议目标

1. 为当前市场日期建立可认证、可版本化的沪深交易日历覆盖；没有日历不得把实时行情升级为研究或执行依据。
2. 定义新闻、公告、财报的最小统一证据模型：Provider/source、官方/原文 URL、公开时间、系统获取时间、正文或文件 Hash、解析版本、股票关联、质量状态与失败原因。
3. 固定每类数据的主 Provider 与只读交叉验证 Provider；禁止 runtime fallback、unknown、Mock 与 Synthetic。
4. 只新增“研究证据观察/审核”能力，输出候选或结论时仍标记为不可交易；不打开 Screener、Backtest 发布、Paper、Live 或 AI 下单。
5. 为后续多维 `Research Readiness` 设计 requirement profile 与 fail-closed 规则；不要把实时 Quote 的 observed 状态传播为 `ready`。

### 推荐任务顺序

```text
Sprint14.3-A：设计评审
  └─ 数据源清单、字段语义、时点模型、失败策略、数据表最小模型、测试与回滚

Sprint14.3-B：认证交易日历覆盖
  └─ 只处理当前受控范围/日期；来源、版本、缺失与 fail-closed 验收

Sprint14.3-C：公告/新闻/财报的只读证据归档试点
  └─ 固定小样本；原文 Hash、公开/获取时间、无 fallback、不可交易标记

Sprint14.3-D：多维研究 Readiness 审核
  └─ 不产生候选、订单或盈利结论，只验证授权键与拒绝路径
```

每个正式子阶段都必须先让用户确认设计，再实施：目标、范围、禁止项、数据源、数据库影响、数据语义、验收、回滚和安全开关。

## 11. 交接结论

可无缝继续，但当前正确的下一步是完善数据时点、证据与研究准入，不是扩大到全市场、运行策略、开放回测/选股或自动模拟交易。任何将实时行情、AI 输出或未认证多维信息直接连接到订单的实现都违反当前已确认的安全与长期设计基线。
