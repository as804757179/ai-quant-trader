# ADR-023：策略版本、审批与有效配置控制面

状态：Accepted

## 决策

内置策略的运行时真相不再来自本地 JSON 文件或 `strategy.strategies.config`。策略管理员只能提交带乐观锁版本号的不可变配置版本；每次提交都会使该策略失去有效配置，直到另一位人工 `risk_admin` 审批当前最新版本。

策略版本保存参数、目录契约 Hash、配置 Hash、提交主体与时间；提交和审批均追加不可变审计事件。审批人不得是提交人，过期版本、目录漂移、Hash 不一致、参数无效或数据库不可用时一律不得成为有效配置。

旧的 create/update 接口保留为兼容提交入口，但必须携带 `expected_revision`，不再直接启用配置。唯一新增的审批端点仅用于旧接口治理修复，不引入新的策略或交易业务能力。

Worker 与可信回测只消费“当前版本且已审批”的数据库快照。异步回测在入队时固化快照，在执行前再次核对当前有效版本；版本变化、撤销或不可用时任务失败关闭。

## 影响

- 未审批、同人审批、陈旧乐观锁提交和目录漂移均不能激活策略。
- `strategy.strategies` 仅保留策略实体与既有关联用途；其旧 `config`、`is_active` 字段不再构成运行时授权。
- 策略变更不改变 Data Certification、回测数据口径、交易权限或六项发布/实盘安全锁。
- 数据库迁移只追加版本、审批和审计结构，不执行破坏性 downgrade。

## 验证与回滚

定向测试覆盖参数与 Hash 校验、乐观锁冲突、同人审批拒绝、陈旧版本拒绝、审批后快照读取、Worker 查询边界与回测执行前复核。迁移和运行时数据库不可用均应 fail closed。

如需暂停该入口，收回 `strategy:write` 或 `strategy:approve` scope；不得删除版本或审批审计记录。
